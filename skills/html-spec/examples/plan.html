Postgres full-text search for candidates

Replace the current ILIKE %query% search with a tsvector-backed search that ranks by relevance and handles typos. Ship behind a feature flag.

TL;DR. Add a generated tsv column to candidates, GIN-index it, swap the search endpoint to use websearch_to_tsquery + ts_rank, ship behind ff_pg_fts. Falls back to ILIKE on the flag-off path.

Effort: 1 dayRisk: lowOwner: backend

Success criteria

  • Search "softwre enginer" returns Software Engineer candidates (typo tolerance via trigram fallback).
  • p95 latency on a 50k-row table stays under 150ms.
  • Ranking surfaces exact-name matches before substring matches.
  • Feature flag flip is reversible without a migration.

Data model

-- migration
ALTER TABLE candidates ADD COLUMN tsv tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('english', coalesce(name, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(headline, '')), 'B') ||
    setweight(to_tsvector('english', coalesce(skills, '')), 'C')
  ) STORED;

CREATE INDEX candidates_tsv_idx ON candidates USING GIN (tsv);
CREATE INDEX candidates_name_trgm_idx ON candidates USING GIN (name gin_trgm_ops);

API change

The existing GET /api/candidates?q= endpoint changes its query body. Response shape stays the same.

// src/app/api/candidates/route.ts
const useFts = await flags.enabled('ff_pg_fts', userId);
const { data } = useFts
  ? await supabase.rpc('search_candidates_fts', { q })
  : await supabase
      .from('candidates')
      .select('*')
      .ilike('name', `%${q}%`);
RPC body (search_candidates_fts) 
CREATE OR REPLACE FUNCTION search_candidates_fts(q text)
RETURNS SETOF candidates AS $$
  SELECT *
  FROM candidates
  WHERE tsv @@ websearch_to_tsquery('english', q)
     OR name % q
  ORDER BY ts_rank(tsv, websearch_to_tsquery('english', q)) DESC,
           similarity(name, q) DESC
  LIMIT 50;
$$ LANGUAGE sql STABLE;

Open questions

Do we need per-tenant ranking weights?

Not for v1. The weight tuple (A=name, B=headline, C=skills) is hardcoded. If recruiters complain that skills should rank higher for technical searches, we can move weights to a user setting later.

What about positions and companies search?

Out of scope. Once this ships and proves out, copy the same pattern to positions and companies in a follow-up PR.
